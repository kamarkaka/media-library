# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

A self-hosted video library web app (like a simpler Jellyfin/Plex). Single-owner. Users add directory paths via Settings, scan for video files, browse/search/filter, and stream with resume-playback support. Compatible files (H.264 + AAC/MP3 in mp4/m4v) play directly via HTTP range requests; everything else is transcoded to HLS on-demand with ffmpeg.

## Commands

- `npm run dev` — start dev server with hot reload (tsx watch), runs on port 3000
- `npm run build` — compile TypeScript to `dist/`
- `npm start` — run compiled output (`node dist/index.js`)
- `npm run setup-auth` — interactive CLI to set login credentials

System dependencies: FFmpeg (`ffprobe` + `ffmpeg`) must be on PATH — `ffprobe` for video metadata extraction during scan, `ffmpeg` for HLS transcoding during playback. Scraping also needs a Chromium binary: the Puppeteer scrapers use `puppeteer-core` (no bundled browser) and launch via `CHROMIUM_PATH` (default `/usr/bin/chromium-browser`, read directly from `process.env` in `src/scrapers/base/browser.ts`, not surfaced in `src/config.ts`). The Dockerfile installs Chromium and sets `CHROMIUM_PATH`; for local scraping install Chromium and set it yourself.

No test framework is configured.

## Architecture

**Server-rendered app**: Express + EJS templates + vanilla JS on the client. No frontend build step — Tailwind is loaded via CDN.

**Database**: SQLite via `better-sqlite3` with Knex query builder. Schema is created imperatively in `src/db.ts` (no migrations directory). New columns are added via `ALTER TABLE` checks at startup, not migrations: a hardcoded `[name, type]` list is reconciled against `PRAGMA table_info` for the `videos` table, plus a separate one-off check that adds `alias` to `genres` — adding a column to any other table needs its own PRAGMA check. Startup also creates indexes (e.g. unique `(run_id, video_id, scraper_type)` on `coverage_results`; indexes on `videos.release_date/director/maker/label` and `playback_state.last_viewed`). Runs in WAL mode with `foreign_keys = ON`. Tables: `videos`, `genres`, `video_genres`, `cast_members`, `video_cast`, `library_paths`, `playback_state`, `settings`, `sessions`, `validation_results`, `coverage_results`, `field_sources`, `scraper_field_config`.

**Auth**: Credentials are stored in the `settings` table (not env vars). On first run, a random session secret and a temporary password are generated; the password (bcrypt-hashed, cost 12) is persisted and printed to console. `npm run setup-auth` (interactive `src/cli.ts`) writes a permanent hash to the DB; `src/cli.ts` also has a `hash-password` subcommand that just prints a hash. Passwords can be changed at runtime via `PUT /api/auth/password` (verifies the current password, re-hashes, updates both the DB and in-memory config — no restart). Sessions use `express-session` with a custom `SQLiteSessionStore` (`src/session-store.ts`); the cookie is httpOnly, `sameSite: 'lax'`, 30-day (`cookie.secure` is not set, so it travels over plain HTTP), with `resave: false` / `saveUninitialized: false`. The store filters expired rows on read but never prunes them. `requireAuth` (`src/middleware/auth.ts`) returns 401 JSON for `/api/*` and redirects to `/login` otherwise.

**Route structure**: Page routes (`src/routes/*.ts`) render EJS views, mounted in `src/index.ts` with mixed prefixes — `libraryRouter` and `tagsRouter` at `/` (`/`, `/genres`, `/cast`), `playerRouter` at `/player`, `settingsRouter` at `/settings`. API routes are aggregated in `src/routes/api/index.ts` and serve JSON under `/api`. All routes except the public login/logout pages (`src/routes/auth.ts`: GET/POST `/login`, POST `/logout`) require session auth via `requireAuth` (`src/middleware/auth.ts`, mounted after the public auth router). Two gotchas: `/api/library` (`src/routes/api/library.ts`) is the **admin/background-jobs router** (scan, scrape, validate, coverage, cover-download, scraper-config, settings, batch-replace, fix-dates, auto-match, db-refresh) — actual video listing lives in `/api/videos`; and `api/tags.ts` and `api/filters.ts` are both mounted at the `/api` root (not `/api/tags` or `/api/filters`), so they expose top-level paths like `/api/genres`, `/api/cast`, `/api/directors`, `/api/search-suggestions` (tags is mounted first, so its `/genres` and `/cast` GET handlers win).

**Background pipeline**: All long-running work runs in Worker threads spawned by `src/services/scanner.ts`, with progress tracked via module-level singletons polled by the frontend. There are four: `scanProgress`, `scrapeProgress`, `coverageProgress`, `coverDownloadProgress`. All four reuse the single `ScanProgress` shape (`status: idle | scanning | done | error` — there is no `'running'` state); each `start*` no-ops if its status is already `'scanning'` (concurrency guard). In dev, `spawnWorker()` constructs an eval Worker that loads `tsx/cjs` first so workers run without a build step.
- **Scan** (`scan-worker.ts`): Walks library paths, discovers video files by extension, inserts new records, removes stale ones, runs `ffprobe` for duration/resolution/codec/bitrate/etc. Quick scan only touches new files; full scan re-probes all. Does not fetch external metadata.
- **Scrape** (`scrape-worker.ts`): Full scrape re-scrapes all videos; quick scrape only processes records with `matched != 1` (a video becomes `matched = 1` once code, name, cover image, and release date are all present). Supports **per-field scraper config** (`scraper_field_config` table) — different DB fields can come from different scrapers (fixed set via `FIELD_TO_META`: code, name, release_date, director, maker, label, cover_image, genres, cast). It groups fields by their configured scraper, pre-creates one resolver+scraper instance per needed scraper type (closed in a `finally` after the whole run, not per-video), and for each video invokes every needed scraper, taking each field from its configured scraper's result. Records which scraper sourced each field in `field_sources` and **skips fields marked `manual`** (batch-loaded once to avoid N+1; applies to scalar fields and the genres/cast relation syncs) so user edits are never overwritten. The per-field config is editable from Settings → Advanced Options and via `GET/PUT /api/library/scraper-config` (empty value deletes the override, falling back to the default scraper). The canonical-ID override is taken only from the default scraper's result. Cover images: with a `code`, `downloadCover` stores a local copy (falls back to the remote URL on failure); without one, the remote URL is stored. The `syncRelation` helper handles many-to-many upserts for genres and cast.
- **Coverage** (`coverage-worker.ts`): Resumable run that reports metadata coverage; results in `coverage_results`. The `run_id` is persisted in the `settings` table under `coverage_run_id` so an incomplete run resumes across restarts (resumability skips `(video_id, scraper_type)` pairs already recorded). Surfaced via `POST /api/library/coverage`, `GET .../coverage/status`, `GET .../coverage/results` (per-scraper success summary for the latest run).
- **Cover download** (`cover-download-worker.ts`): Backfills missing cover images.

**Streaming** (`src/services/hls-transcoder.ts`, `src/routes/api/videos.ts`): `player.ts` computes `canDirectPlay` (H.264 + AAC/MP3 + mp4/m4v). Compatible files stream raw via `GET /api/videos/:id/stream` (HTTP range requests). Otherwise the hls.js client requests `GET /api/videos/:id/hls` (master), `/:quality` (variant, triggers ffmpeg transcode), and `/:quality/:segment`. Qualities are 360p/720p/1080p (only those ≤ source height) plus `original`, which is always offered as the highest-bandwidth level regardless of source height (stream-copy when already H.264 + AAC/MP3, else re-encoded with libx264/aac). The variant endpoint rewrites bare `seg\d+.ts` lines in the on-disk `playlist.m3u8` to absolute URLs; `startTranscoding` resolves only once `playlist.m3u8` appears (rejects if ffmpeg exits without it or after a 60s timeout). Segments cached under `HLS_CACHE_DIR/<videoId>/<quality>/`; active jobs deduped in a `Map`; `cleanupCache(videoId)` removes a video's cache dir. The client is a **custom player** (`public/js/player.js` + `views/player.ejs`): source-selection fallback chain (directPlay → hls.js → native HLS → raw stream), quality selector driven by hls.js levels (Auto/-1), custom seek bar, fullscreen (incl. iOS `webkitEnterFullscreen`), keyboard shortcuts, swipe-to-seek (step from the `seek_step` setting), and a click-to-play cover overlay.

**Scraper plugin system** (`src/scrapers/`): Scrapers are loaded dynamically by directory name (loader `src/scrapers/base/index.ts`); failures fall back to `NoOpScraper`. To add a new scraper, create `src/scrapers/<name>/` with:
- `scraper.ts` — must export `createScraper(): Scraper` (required). Browser-based scrapers can extend the `PuppeteerScraper` base (`base/puppeteer-base.ts`) and implement `buildUrl()` + `extractMetadata(page)`.
- `config.ts` — per-scraper constants like `SEARCH_URL_PREFIX` / `SOURCE_URL_PREFIX` (convention used by existing scrapers)
- `resolver.ts` — must export `resolveSourceUrl(filename) → URL`; optional `closeResolver()` (optional)
- `validator.ts` — must export `getTestConfig(): ValidatorTestConfig` for automated validation (optional)

Existing scrapers: `javtrailers` (default), `123av`, `javxx`, `missav`, `javfilms` — all five ship a `validator.ts`. The loader special-cases `noop` (and invalid/empty names) → `NoOpScraper`, and `listScrapers()` auto-discovers available scrapers by scanning the directory (this is how the UI enumerates them, not a hardcoded list). The active scraper is set via `SCRAPER_TYPE` env var. Base types are in `src/scrapers/base/types.ts`.

**Validator cron** (`src/services/validator-scheduler.ts`): On boot (unless `VALIDATOR_ENABLED=false`), schedules `runAllValidations()` on `VALIDATOR_CRON` (default daily 08:00). Each run resolves + scrapes a known test file per scraper, compares against the validator's `expected`, and stores rows in `validation_results` (pruned to a global cap of `30 × (scraper count)` most-recent rows by `created_at` — an aggregate cap, not strictly 30 per scraper).

**Video queries** (`src/services/video-queries.ts`): Centralized query builder for filtering, sorting, pagination, and playback state lookups — shared by both page routes and API routes. Library browsing uses multi-select **pill filters**: `parseVideoFilters` splits comma-separated query params (`?genre=Action,Drama&cast=Alice,Bob&match=unmatched`) into arrays for genre/director/maker/label/cast plus a `match` filter keyed off the `videos.matched` boolean. Sorting has two effective modes only: `last_viewed` (joins `playback_state`, DESC NULLS LAST) and everything else (release_date ASC NULLS LAST). Also exports `getVideoNeighbors` (prev/next for player navigation) and filter-option lookups for the filter UI.

## Key Patterns

- Config is centralized in `src/config.ts`, sourced from env vars / `.env`. Vars: `PORT` (3000), `DB_PATH` (`data/library.db`), `SCRAPER_TYPE` (`javtrailers`), `VALIDATOR_ENABLED` (true unless `'false'`), `VALIDATOR_CRON` (`0 8 * * *`), `HLS_CACHE_DIR` (`data/hls-cache`), `FFMPEG_PATH` (`ffmpeg`), `COVER_CACHE_DIR` (`data/covers`). Default data paths are module-relative (`path.join(__dirname, '../data/...')`), so they land under `src/../data` in dev and `dist/../data` after build. `CHROMIUM_PATH` (scrapers) and `NODE_ENV` (disables Express view cache when not `production`) are read from `process.env` directly, not via `config.ts`. Note: `.env.example` ships `SCRAPER_TYPE=noop`, so copying it gives no-op scraping by default even though the code default is `javtrailers`.
- Auth credentials and the session secret are not env vars — they live in the `settings` table and are loaded into config at boot. The `settings` table also holds runtime keys editable from the UI without restart: `seek_step` (5/10/15/30), `default_scraper`, and `coverage_run_id`.
- Video IDs are UUIDs, but scrapers can override them with a canonical ID
- User edits are never lost: `PUT /api/videos/:id` records each edited field in `field_sources` as `source='manual'` (or the originating scraper when applying scrape results), the scrape worker skips `manual` fields, and the player shows per-field source badges. (`source_url` is a legacy column in `src/db.ts` — no longer surfaced or written.)
- Playback position is saved per-video and used for resume functionality
- Worker threads prevent concurrent runs via status check on the progress singleton
- Docker support via `Dockerfile` and `docker-compose.yml`: two-stage Node 20 Alpine build (`npm ci`, needs a committed `package-lock.json`), installs ffmpeg + chromium, sets `CHROMIUM_PATH` and `DB_PATH=/data/library.db`, declares a `/data` volume; `docker-compose.yml` mounts `./data:/data` and expects videos at `/media:ro` (edit `/path/to/your/videos`)
